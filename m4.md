## M4-A: Incident Creation with PostGIS Location
**Priority:** High | **Status:** ✅ Complete | **Issue #48**

### Scope
Regional Encoder creates incidents with geometry point.

### What exists in the codebase
- **Backend:** `POST /api/regional/incidents` — `create_incident()` in `api/routes/regional.py`. Uses `ST_SetSRID(ST_MakePoint(:lon, :lat), 4326)`. Region isolation enforced via `get_regional_encoder` dependency + `region_id` from token.
- **Frontend:** `/incidents/create` page renders `<IncidentForm />` with `<MapPicker />` component. Lat/lon fields + map picker both present. Submits to backend via `apiFetch`.
- **Regional list:** `GET /api/regional/incidents` returns incidents scoped to the encoder's region.

### Exit Criteria
- [x] Encoder can create incident with map-picked coordinates
- [x] Incident appears in regional incident list

## M4-B: Incident Edit (Own, Non-Verified Only)
**Priority:** High | **Status:** 🟡 Partial | **Issue #49**

### What exists in the codebase
- **Backend:** `PUT /api/regional/incidents/{id}` — `update_incident()` exists and updates `incident_nonsensitive_details` and `incident_sensitive_details`, with PII re-encryption on change.
- **Frontend:** `/dashboard/regional/incidents/[id]` page exists — but it is a **read-only detail view only**. There is no edit form or edit mode on this page.

### ⚠️ Issues Found

**Issue 1 — Frontend edit mode missing.**
The `/dashboard/regional/incidents/[id]` page (`app/dashboard/regional/incidents/[id]/page.tsx`, 439 lines) renders a structured detail view with no edit controls, no form inputs, and no call to `PUT /api/regional/incidents/{id}`. The backend endpoint exists but the frontend never calls it.

**Issue 2 — Status guard mismatch with spec.**
`update_incident()` allows edits when `verification_status IN ('DRAFT', 'PENDING', 'REJECTED')`. The M4 spec states only `PENDING_REVIEW` incidents should be editable. The codebase uses `PENDING` (not `PENDING_REVIEW`) as the submitted status, and also permits edits on `DRAFT` and `REJECTED` — wider than the spec intends.

**Issue 3 — No per-encoder ownership check.**
`update_incident()` only verifies `region_id` matches. It does **not** verify `encoder_id = current_user`. Any encoder in the same region can edit any other encoder's incident. The spec requires: `encoder_id == current_user`.

**Issue 4 — Edit does not create an audit trail entry.**
`update_incident()` updates the row and calls `db.commit()` with no `INSERT INTO wims.incident_verification_history`. The M4 spec requires every edit to produce an audit log entry.

### Exit Criteria
- [x] Encoder can edit their own `PENDING_REVIEW` incidents _(backend exists; frontend edit UI **missing**)_
- [x] Encoder cannot edit `VERIFIED` incidents _(partial — VERIFIED is blocked, but same-region cross-encoder edits are not blocked)_
- [ ] Edit creates audit trail entry

## M4-D: Duplicate Detection on Import
**Priority:** High | **Status:** 🔴 Incomplete | **Issue #51**

### What exists in the codebase
- **Frontend only:** `/afor/import` page calls `window.confirm()` with "Replace Original / Keep Original" when a `409` response is returned from the commit endpoint. This is a global all-or-nothing decision, not a per-row modal.
- **Backend:** The commit endpoint returns a `409` on duplicate, but there is **no spatial `ST_DWithin` query** executed before commit. Duplicate detection logic in `regional.py` does not exist.

### ⚠️ Issues Found

**Issue 5 — No spatial pre-commit duplicate scan.**
The M4 spec requires a `ST_DWithin` query (1 km radius) against `(region_id, incident_date, alarm_level)` before committing. This does not exist in the backend. The 409 response the frontend handles appears to come from DB constraint violations, not intentional pre-check logic.

**Issue 6 — No per-row skip/merge/force UI.**
The frontend resolves duplicates with a single `window.confirm()` for the entire batch ("Replace All" or "Keep All"). The spec requires a modal showing each duplicate with per-row options: skip, merge, or force-create.

### Exit Criteria
- [ ] Import shows duplicate confirmation before commit _(global confirm only, not per-row modal)_
- [ ] Encoder can choose skip/merge/force per duplicate _(binary replace/keep only)_

## M4-E: Draft Save
**Priority:** Medium | **Status:** 🟡 Partial | **Issue #52**

### What exists in the codebase
- All incidents created via `POST /api/regional/incidents` or `POST /api/regional/afor/commit` default to `verification_status = 'DRAFT'`.
- The validator queue (`GET /api/regional/validator/incidents`) filters to `PENDING` and `PENDING_VALIDATION` by default, so DRAFTs are naturally hidden from validators.
- **No dedicated draft endpoints exist:** `POST /api/regional/incidents/draft`, `GET /api/regional/incidents/drafts`, `PATCH /api/regional/incidents/draft/{id}` — none of these are implemented.
- **No Celery expiry task** for 30-day draft auto-expiry (no task in `tasks/` references draft expiry).
- **No frontend draft-specific UI** — no save-as-draft button, no drafts list, no resume flow.

### ⚠️ Issues Found

**Issue 7 — No dedicated draft CRUD endpoints.**
Drafts are implicitly created but cannot be explicitly managed. Encoders have no way to list, update, or delete drafts separately from the main incident list.

**Issue 8 — No draft auto-expiry task.**
The Celery periodic task for 30-day draft expiry described in the spec does not exist.

### Exit Criteria
- [x] Drafts do not appear in validator queue _(DRAFTs are filtered out by default status filter)_
- [ ] Encoder can save and resume drafts _(no explicit draft save/resume flow)_

## M4-F: National Validator Verification Queue
**Priority:** High | **Status:** 🟡 Partial | **Issue #53**

### What exists in the codebase
- **Backend:** `GET /api/regional/validator/incidents` — protected by `get_national_validator`. Supports `status`, `encoder_id`, `limit`, `offset`, and `show_all` filters. Inserts audit trail on every verify action.
- **Backend:** `PATCH /api/regional/incidents/{id}/verification` — `verify_incident()`. Supports `accept → VERIFIED`, `pending → PENDING`, `reject → REJECTED`. Inserts into `wims.incident_verification_history` in the same transaction. Idempotency guard (409 if already at target status).
- **Frontend:** `/dashboard/validator` page — full queue UI with status filter tabs, encoder filter input, pagination, action modal with notes field (accept/pending/reject).

### ⚠️ Issues Found

**Issue 9 — Queue is region-scoped, not cross-region.**
The M4 spec states the validator queue should return incidents **cross-region** (`all PENDING_REVIEW`). The actual implementation enforces `fi.region_id = :region_id` (the validator's own `assigned_region_id`). National validators cannot see incidents from other regions.

**Issue 10 — Status name is `PENDING`, not `PENDING_REVIEW`.**
The codebase uses `PENDING` and `PENDING_VALIDATION` as the submitted statuses. The spec consistently refers to `PENDING_REVIEW`. This inconsistency affects queue filtering and guard logic across M4-B and M4-F.

### Exit Criteria
- [ ] Validator sees cross-region queue _(currently region-scoped only)_
- [x] Validator can approve or reject with reason
- [x] Each action creates audit trail entry _(confirmed: `incident_verification_history` insert in `verify_incident()`)_

> _The audit trail criterion was marked `[ ]` in the original milestone doc but is confirmed implemented._
